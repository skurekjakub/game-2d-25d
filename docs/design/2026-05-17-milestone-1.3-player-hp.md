# Milestone 1.3 — Player HP, Contact Damage, Run-end on Death

**Date:** 2026-05-17
**Status:** approved (brainstorming complete)
**Predecessor:** [milestone-1.2 spawner](../plans/2026-05-17-milestone-1.2-spawner.md)
**Parent design:** [vampire-survivors-clone-design](2026-05-16-vampire-survivors-clone-design.md)

## Goal

Close the combat loop in the other direction: enemies can now kill the player. At end of M1.3 the game has a fail state — touch enemies long enough and the run ends.

## Scope

In:
- `ContactDamageComponent` on each enemy, tick-damaging the Player on overlap.
- `Hurtbox` (Area2D) on each enemy as the detection shape.
- `HealthComponent` reused on the Player (moved out of the enemy-only namespace).
- `EventBus.run_ended(false)` emit when Player HP hits zero.
- Sprite-flash visual feedback on the Player when damaged.
- RunState refactor: `hp` / `max_hp` removed (component-owned now, not run-state-owned).

Out (deferred to later milestones):
- Game Over screen / restart UI.
- HP HUD bar.
- i-frames / invulnerability windows (intentional — DPS-tick stacking IS the lethality model).
- Knockback / hit-stun on the Player.
- Audio.
- Enemy variety with different `contact_damage` values (basic_walker only for now).

## Key decisions (locked in brainstorming)

| # | Decision | Choice |
|---|---|---|
| 1 | Damage cadence | DPS tick, 1.0s interval, fire-on-enter, reset-on-exit |
| 2 | Detection topology | Hurtbox `Area2D` on each enemy detects Player's CharacterBody2D |
| 3 | HP source of truth | `HealthComponent` on Player; RunState loses `hp`/`max_hp` |
| 4 | Death behavior | Soft stop — emit `run_ended(false)`, Player goes inert, Arena keeps running |
| 5 | Damage feedback | Sprite flash (red tint, ~0.1s) on Player on `damaged` signal |
| 6 | i-frames | None — stacking damage from blobs is intended |

## Architecture

```
Player (CharacterBody2D)
├── CollisionShape2D                # existing — body collision for movement
├── Visual (Polygon2D)              # existing — flash target
├── WeaponHost                      # existing — disabled on death
└── HealthComponent  ← NEW          # max_hp=100; emits damaged / died

Enemy (CharacterBody2D)
├── CollisionShape2D                # existing — body collision (separates enemies)
├── Visual (Polygon2D)              # existing
├── HealthComponent                 # existing — takes weapon damage
├── WalkTowardPlayerComponent       # existing — AI
├── Hurtbox (Area2D)  ← NEW         # detects Player overlap
│   └── CollisionShape2D            # CircleShape2D, radius ~12 (matches body)
└── ContactDamageComponent ← NEW    # owns the per-player tick timer
```

### `combat/components/health.gd` (relocated from `combat/enemies/components/health.gd`)

No behavioral change. The class is general-purpose; the path was historically wrong because the only user was the enemy. Move now, before adding a second user on the Player. The single existing `[ext_resource path="res://combat/enemies/components/health.gd"]` reference in `combat/enemies/enemy.tscn` updates accordingly. No `Array[HealthComponent]` exists in any `.tres`, so the typed-Array path-fragility issue ([[godot_typed_resource_array_path_fragility]]) does not apply — this is a `class_name`'d Node script, not a Resource referenced by typed Array.

### `combat/components/contact_damage.gd` — new

```gdscript
class_name ContactDamageComponent
extends Node

## Ticks `damage` to overlapping bodies once per `tick_interval`. Bind the
## hurtbox via `hurtbox_path`; the component reads body_entered / body_exited
## from it. Per-body timers stored in a dict — entering resets, exiting clears.

@export var damage: float = 5.0
@export var tick_interval: float = 1.0
@export var hurtbox_path: NodePath           # [[godot_node_export_nodepath_resolution]] — NodePath, resolved in _ready

var _hurtbox: Area2D
var _timers: Dictionary = {}                 # { body_instance_id (int): seconds_until_next_tick (float) }

func _ready() -> void:
    _hurtbox = get_node_or_null(hurtbox_path) as Area2D
    assert(_hurtbox != null, "ContactDamageComponent: hurtbox_path must resolve to an Area2D")
    _hurtbox.body_entered.connect(_on_body_entered)
    _hurtbox.body_exited.connect(_on_body_exited)

func _process(delta: float) -> void:
    if _timers.is_empty():
        return
    for body_id in _timers.keys():
        _timers[body_id] -= delta
    # Second pass: ticks + cleanup of freed bodies. (Don't mutate while iterating above.)
    for body_id in _timers.keys():
        if _timers[body_id] > 0.0:
            continue
        var body: Node = instance_from_id(body_id) as Node
        if body == null or not is_instance_valid(body):
            _timers.erase(body_id)
            continue
        _apply_tick(body)
        _timers[body_id] = tick_interval

func _on_body_entered(body: Node) -> void:
    if not _is_damageable(body):
        return
    # Fire-on-enter: immediate first tick, then schedule next.
    _apply_tick(body)
    _timers[body.get_instance_id()] = tick_interval

func _on_body_exited(body: Node) -> void:
    _timers.erase(body.get_instance_id())

func _is_damageable(body: Node) -> bool:
    # A body is damageable iff it has a HealthComponent child.
    return _find_health(body) != null

func _apply_tick(body: Node) -> void:
    var health: HealthComponent = _find_health(body)
    if health == null:
        return
    health.take_damage(damage, get_parent())   # source = the enemy (parent of this component)

static func _find_health(body: Node) -> HealthComponent:
    for child in body.get_children():
        if child is HealthComponent:
            return child
    return null
```

**Why per-body timers in a Dict and not a `Timer` node:** a single enemy can theoretically overlap multiple damageable bodies (today only the Player, but the cleanup pattern shouldn't assume that). `Timer` is one-shot per node; dict-of-deltas scales.

**Why `instance_from_id` round-trip:** holding the `Node` reference directly would keep a freed body alive in the dict; the int id is safe to store and `instance_from_id` returns null for freed instances, giving us free cleanup of bodies that despawn mid-overlap.

**`_is_damageable` check:** keeps the component agnostic — it doesn't know about `Player` as a class. Any body with a `HealthComponent` child is fair game. This lets us later add a destructible obstacle without touching the component.

### `combat/enemies/enemy.tscn` — modified

Adds two children to the existing root node:

```
[node name="Hurtbox" type="Area2D" parent="."]
collision_layer = 0          # doesn't need to be detected by others
collision_mask = <PLAYER_LAYER>   # only scans for player

[node name="CollisionShape2D" type="CollisionShape2D" parent="Hurtbox"]
shape = SubResource("CircleShape2D_enemy_hurtbox")   # radius 12 (slightly larger than the 10 body)

[node name="ContactDamageComponent" type="Node" parent="."]
script = ExtResource("contact_damage")
damage = 5.0
tick_interval = 1.0
hurtbox_path = NodePath("../Hurtbox")
```

The hurtbox radius is intentionally slightly larger than the body (12 vs 10) so contact damage triggers a frame before bodies clip — feels fair, prevents zero-velocity grazing edge cases. `damage = 5.0` is overridden per-enemy at runtime by `enemy.gd._apply_data()` reading `EnemyData.contact_damage`, same pattern as it already does for `HealthComponent.max_hp`.

### `player/player.gd` — modified

```gdscript
@onready var _health: HealthComponent = $HealthComponent
@onready var _weapon_host: Node = $WeaponHost
@onready var _visual: Polygon2D = $Visual

const FLASH_COLOR := Color(1.0, 0.3, 0.3, 1.0)
const FLASH_DURATION_SEC := 0.1

var _is_dead: bool = false
var _base_color: Color


func _ready() -> void:
    _base_color = _visual.color
    _health.damaged.connect(_on_damaged)
    _health.died.connect(_on_died)


func _on_damaged(amount: float, _new_hp: float) -> void:
    EventBus.damage_dealt.emit(null, self, amount)   # source unknown at this layer; enemy-as-source is set by ContactDamage when relevant
    _flash()


func _on_died(_killer: Node) -> void:
    if _is_dead:
        return
    _is_dead = true
    velocity = Vector2.ZERO
    set_physics_process(false)
    _weapon_host.set_process(false)
    _weapon_host.set_physics_process(false)
    EventBus.run_ended.emit(false)


func _flash() -> void:
    _visual.color = FLASH_COLOR
    var tween := create_tween()
    tween.tween_property(_visual, "color", _base_color, FLASH_DURATION_SEC)
```

**`source` argument in `damage_dealt`:** ContactDamageComponent passes `get_parent()` (the enemy) into `take_damage(amount, source)`, but `HealthComponent.damaged` doesn't carry the source. The Player-side handler emits `damage_dealt(null, self, amount)` as a deliberate compromise — the source isn't known at the HealthComponent boundary, and threading it through would require widening the `damaged` signal (out of scope, breaks Enemy compat). The debug log will show `target=Player, source=null`. Acceptable for M1.3; a later HUD can derive source from spatial proximity if it needs to.

**`_is_dead` guard:** belt-and-suspenders against `died` firing twice (HealthComponent already guards with `_died_emitted`, but defense-in-depth here is one line).

**Stopping `_weapon_host.set_physics_process(false)`:** WeaponHost ticks weapon timers; flipping both flags freezes auto-fire. The weapon nodes themselves are unchanged — projectiles already in flight continue (correct behavior, you can still see your last shots land after death).

### `bootstrap/game.gd` — refactor

Remove `hp` and `max_hp` from `RunState`. Remove the two `run_state.max_hp = ...` / `run_state.hp = ...` lines from `start_run()`. RunState becomes:

```gdscript
class RunState:
    var xp: int
    var level: int
    var time_elapsed: float
```

`start_run()` keeps `xp = 0`, `level = 1`, `time_elapsed = 0.0`, and the `run_started` emit. Player HP is the HealthComponent's concern, initialized on its own `_ready` from the `@export max_hp` set in the scene.

### `bootstrap/debug.gd` — modified

Add a `run_ended` subscription that logs `[EventBus] run_ended(victory=false)`. (`damage_dealt` already subscribed per the M1.1 work; that channel now fires for the Player too.)

## Data flow

### Damage path (Player getting hurt)

```
1. Enemy hurtbox detects Player CharacterBody2D
   → body_entered(player)
2. ContactDamageComponent._on_body_entered(player)
   - finds HealthComponent on player
   - applies first tick immediately (fire-on-enter)
   - stores instance_id in _timers with tick_interval seconds left
3. Player.HealthComponent.take_damage(5.0, enemy)
   - hp -= 5
   - emits damaged(5.0, new_hp)
   - if hp == 0 and not _died_emitted: emits died(enemy)
4. Player._on_damaged(5.0, new_hp)
   - emits EventBus.damage_dealt(null, self, 5.0)
   - tweens Visual.color red → base over 0.1s
5. Debug logs "[EventBus] damage_dealt(... target=Player, amount=5)"
6. Each subsequent _process tick:
   - timer counts down; when ≤0, applies tick again, resets to 1.0
7. On exit: ContactDamageComponent._on_body_exited(player) erases timer
```

### Death path

```
1. Player.HealthComponent.hp hits 0 → emits died(enemy)
2. Player._on_died(enemy):
   - velocity = Vector2.ZERO
   - set_physics_process(false)   ← Player stops moving
   - WeaponHost processes disabled  ← auto-fire stops
   - emits EventBus.run_ended(false)
3. Debug logs "[EventBus] run_ended(victory=false)"
4. Arena keeps running (enemies keep walking, spawner keeps ticking).
   Player visually freezes in place. Smoke test: this is the visible end state.
```

## Edge cases

- **Player despawns mid-overlap** (e.g., future scene change): `instance_from_id` returns null → timer erased silently. No crash.
- **Enemy despawns mid-overlap** (death from projectile while touching Player): the enemy node and its ContactDamageComponent are freed; the per-component timer dict goes with them. `body_exited` may or may not fire depending on `monitoring` state at queue_free — same Area2D-pickup race as XpGem. **Apply the [[godot_area2d_pickup_pattern]]**: enemy's `_die()` (or equivalent) must `Hurtbox.set_deferred("monitoring", false)` before `queue_free()`, mirroring the XpGem fix. Without this, the player could take one extra phantom tick on the same frame the enemy dies.
- **Multiple enemies on same frame at t=0**: each contributes its own first tick immediately. If 5 walkers spawn on top of a stationary player (won't happen with the M1.2 fix that spawns off-camera, but hypothetically), Player takes 25 dmg on frame 1, then 25 dmg/s after. Intended.
- **Re-entrancy in HealthComponent**: existing `_died_emitted` guard from M1.1 covers this. Player's `_is_dead` is the second layer.
- **Damaged signal during a tween**: each `_flash()` calls `create_tween()` which makes a fresh tween. If damage arrives mid-flash, the new tween supersedes the old (Godot Tweens default to autostart and don't block each other on the same property — the latest tween wins on the property). Acceptable: visually the flash just looks continuous through rapid damage.
- **`died` while `_physics_process` was already off** (theoretical, can't happen in M1.3): the guard handles it.
- **add_child during body_entered chain**: ContactDamageComponent does not spawn nodes (only mutates HealthComponent state and a dict). The signal-callback-add-child gotcha ([[godot_signal_callback_addchild]]) does not apply here.

## Testing strategy

### Unit (GdUnit4)

**`tests/unit/test_contact_damage_component.gd`** — new
- `test_first_tick_fires_on_enter`: stub body with HealthComponent, simulate `body_entered`, assert HP decreased by `damage` immediately.
- `test_ticks_at_interval`: drive `_process(delta=tick_interval)` advances, assert each tick deducts `damage`.
- `test_no_tick_after_exit`: enter, exit, advance time, assert no further damage.
- `test_exit_then_reenter_resets_timer`: enter, exit before tick fires, re-enter, assert first tick fires immediately again (not on residual timer).
- `test_non_damageable_body_ignored`: body without HealthComponent → `body_entered` is a no-op.
- `test_freed_body_cleanup`: enter, free the body, drive `_process`, assert no crash and timer is dropped.

**`tests/unit/test_health.gd`** — relocate / update path reference (no behavioral change). Existing tests stay green.

**`tests/unit/test_game_run_state.gd`** — remove any assertion on `run_state.hp` / `run_state.max_hp`. Confirm `start_run()` still resets xp/level/time_elapsed.

### Integration (headless)

`godot --headless --quit --path .` should still exit 0 and produce only the existing M1.2 expected lines. No new headless smoke output for M1.3 (the death path needs visual confirmation).

### Visual smoke test (one-time, before tagging)

1. Run Arena. Stand still in the middle.
2. Wait for an enemy to reach you. Confirm:
   - Player flashes red the moment the enemy touches.
   - Console logs `[EventBus] damage_dealt` lines once per second per overlapping enemy.
   - Player visually flashes once per damage tick.
3. Let multiple enemies pile on. Confirm damage stacks (HP drains faster — verify via console line count, since no HUD).
4. Continue until HP hits zero. Confirm:
   - Player stops moving.
   - Auto-fire stops (no new projectiles).
   - Console logs `[EventBus] run_ended(victory=false)` exactly once.
   - Existing projectiles in flight continue and resolve.
   - Spawner keeps spawning enemies (Arena does not pause).
5. Press Esc / kill the process. (No restart UI in M1.3.)
6. All M1.0–M1.2 behaviors still work (movement, auto-fire pre-death, projectile damage to enemies, gem pickup, wall collision, F3 debug toggle).

## Risk register

| Risk | Mitigation |
|---|---|
| Moving `health.gd` breaks `enemy.tscn` `[ext_resource]` path | Single-file path update; verified by headless smoke test (Godot errors loudly on missing ext_resource). The `class_name HealthComponent` stays the same, so any code-side `is HealthComponent` keeps working. |
| `RunState.hp` removal breaks an unknown caller | Grep the repo for `run_state.hp` / `run_state.max_hp` during plan-writ; every site shows up. Done at plan time. |
| Hurtbox vs body collision-layer mismatch leaves enemies non-damaging | First task in plan should be the layer audit (Player layer, enemy body layer, enemy hurtbox mask). Visual smoke test would catch but expensive — better to lock layers explicitly in the plan. |
| `body_exited` race on enemy death mid-overlap | Apply `set_deferred("monitoring", false)` to the Hurtbox in enemy death path (mirrors XpGem fix). Explicit task in plan. |
| Tween-on-flash leaks if damage rate is huge | Tweens auto-cleanup when the property reaches target or another tween supersedes. No leak. |
| Player has no source-of-damage info in `damage_dealt` emit | Documented compromise; null source acceptable for M1.3 (no UI consumes it). Future HUD can derive from spatial proximity if needed. |
