# Milestone 1.4 — HUD + Level-up Modal + Upgrade System Foundations

**Date:** 2026-05-17
**Status:** approved (brainstorming complete)
**Predecessor:** [milestone-1.3 player-hp](2026-05-17-milestone-1.3-player-hp.md)
**Parent design:** [vampire-survivors-clone-design](2026-05-16-vampire-survivors-clone-design.md)

## Goal

Make the game playable as a *complete short run*. Surface invisible state via a HUD so the player can see their HP, XP, level, time, weapons. Wire `EventBus.level_up` into a 3-card upgrade-pick modal that pauses the game, lets the player pick 1 of 3 random upgrades, and applies the upgrade by appending to `RunState.upgrades_taken`. Extend the spawn schedule to a 10-minute run with 1-minute phase ramps. Emit `EventBus.run_ended(true)` when the schedule completes.

## Scope

In:
- HUD (5 elements): HP bar, XP bar, level number, run timer (mm:ss), weapons list (text), FPS readout.
- Level-up modal: triggered by `EventBus.level_up`, pauses game, shows 3 random upgrades, click-or-key to pick.
- Upgrade data system: `UpgradeData` Resource, 5 `.tres` files (4 stats + 1 heal), `UpgradeRegistry` autoload, pure-functional application via `RunState.upgrades_taken`.
- `WeaponInstance.effective_damage()` / `effective_fire_rate()` walk `RunState.upgrades_taken`.
- Player `max_hp` / `move_speed` upgrades mutate Player + HealthComponent directly (per-instance state is fine to mutate; data templates are not).
- Spawn schedule extended to 10 phases × 60s = 10-min full run.
- `EventBus.run_ended(true)` emitted by SpawnerDirector at schedule end.

Out (deferred):
- Weapon icons (placeholder text "Blaster Lv.1" only).
- Game Over / Victory UI screens — `run_ended` just fires; debug.gd logs it.
- Restart button.
- Upgrade weights / rarities (uniform random pick across pool).
- Mouse-hover tooltips with extended descriptions.
- Reroll / skip / banish upgrade slot mechanics.
- Save / load (Phase 2).
- Audio.
- Theming — Godot defaults for all UI nodes.

## Key decisions (locked in brainstorming)

| # | Decision | Choice |
|---|---|---|
| 1 | Scope | HUD + level-up modal (both, bundled milestone) |
| 2 | HUD elements | All 5: HP / XP / level / timer / weapons-list (text) / FPS |
| 3 | Upgrade pool | 4 stat upgrades (+max HP, +move speed, +weapon damage, +fire rate) + 1 heal-to-full |
| 4 | Pool repeats | Allowed (compounding stacks — standard VS) |
| 5 | Upgrade architecture | Option 3: pure-functional. `RunState.upgrades_taken` is the source of truth; weapons compute effective values from base + upgrade list at fire time. Player stats mutate the Player directly (per-instance), but the upgrade entry still goes in the list for audit. |
| 6 | Modal input | Mouse + keyboard (1/2/3, Tab/arrow + Enter, click) |
| 7 | Multi-level queue | Queued back-to-back. One modal per level crossed. |
| 8 | Run length | 10 min / 10 phases × 60s ramps |
| 9 | Victory emit | `EventBus.run_ended(true)` from SpawnerDirector when schedule's last phase completes |

## Architecture

### Data flow at fire time (read path)

```
WeaponHost._physics_process(delta):
  for weapon in weapons:
    weapon.tick(delta)
    if weapon.can_fire():
      weapon.reset_cooldown()        # uses effective_fire_rate()
      spawn projectile with damage = weapon.effective_damage()

WeaponInstance.effective_damage():
  d := data.damage
  for upgrade in Game.run_state.upgrades_taken:
    if upgrade.id == "weapon_damage_25":
      d *= 1.25
  return d
```

The walk happens once per shot. At realistic pool sizes (5 upgrades in M1.4, ~30 in late-game) this is a few microseconds per call — perf-irrelevant. Always reads current state; no stale multiplier fields to keep in sync.

### Data flow on level-up (write path)

```
1. Player picks up XpGem → Game.add_xp(value) → run_state.xp += value
2. While run_state.xp >= xp_needed(level):
     run_state.level += 1
     EventBus.level_up.emit(new_level)
3. UpgradePicker._on_level_up(new_level):
     _queue.append(new_level)
     if not _modal_open:
       _show_next_in_queue()
4. UpgradeModal opens:
     get_tree().paused = true   (modal has process_mode = PROCESS_MODE_WHEN_PAUSED)
     options = UpgradeRegistry.pick_random_3()
     render 3 cards
5. Player picks card N:
     upgrade = options[N]
     UpgradeRegistry.apply(upgrade, player)
     run_state.upgrades_taken.append(upgrade)
     close modal
     if _queue.size() > 0:
       _show_next_in_queue()  # next level's modal
     else:
       get_tree().paused = false
```

`UpgradeRegistry.apply` mutates per-instance state (Player.speed, HealthComponent.max_hp, HealthComponent.hp). Weapon-related upgrades are no-op at apply time — they take effect via the `effective_*()` read path that walks `upgrades_taken`. This means weapon upgrades are "instant" without touching any weapon code at apply.

### Why `UpgradeRegistry` is an autoload (not a Node)

Same reason as `Game` and `EventBus`: it's a global query surface (pick_random anywhere; apply from the modal). No per-scene state. Owns the loaded UpgradeData pool. Autoload count goes from 3 to 4 — within the project's approved list (rubber-duk knowledge base would flag a 5th without justification).

### UpgradeData shape

```gdscript
class_name UpgradeData
extends Resource

@export var id: StringName              # stable id for switch dispatch ("max_hp_20")
@export var display_name: String        # "+20 Max HP (+heal)"
@export var description: String         # one-line flavor / detail shown on card
@export_range(0.0, 10.0) var weight: float = 1.0   # weighted random; default uniform
```

`UpgradeRegistry.apply(upgrade, player)` is a `match upgrade.id:` switch. Adding upgrade #20 in M2.7 = drop a `.tres` + add one `match` branch. Simple linear scaling.

### Pause mechanism

`get_tree().paused = true` halts every node by default. The modal opts back in with:

```
[node name="UpgradeModal" type="Control" ...]
process_mode = 3   # PROCESS_MODE_WHEN_PAUSED
```

This is the documented Godot pause pattern. Crucially, projectiles-in-flight, enemies-mid-walk, the spawner, and the Player all freeze cleanly — no half-state. Audio doesn't either yet (no audio system). When the modal closes, `paused = false` and physics resumes from exactly where it was.

### HUD subscriptions

```
HUD._ready():
  EventBus.run_started.connect(_on_run_started)         # reset visible state
  EventBus.run_ended.connect(_on_run_ended)             # fade out / show "Run Over"
  EventBus.damage_dealt.connect(_on_damage_dealt)       # may update HP bar (only if target is Player)
  EventBus.xp_gained.connect(_on_xp_gained)             # update XP bar
  EventBus.level_up.connect(_on_level_up)               # update level label
  # HP/XP bars also poll Game.run_state in _process for catch-up (signals may miss the first frame)

HUD._process(delta):
  _timer_label.text = format_mm_ss(Game.run_state.time_elapsed)
  _fps_label.text = "%d fps" % Engine.get_frames_per_second()
```

Mix of signal-driven (event-shaped state) and poll-driven (continuous values like time elapsed and FPS). HP bar is signal-driven (`damage_dealt`) but also reads `player._health.hp` in `_process` for max-hp-changed events (no signal fires for that).

### Weapons list element

Iterates `player._weapon_host.weapons` each frame, renders one line per weapon:
```
Blaster Lv.1
```

`weapon.level` is the count of `weapon_damage_25` / `fire_rate_30` upgrades taken — a derived number, not stored. (When we have multiple weapons in M1.5+, this becomes one line per weapon.)

### Schedule extension

Extend `world/data/wave_01_schedule.tres` from 2 phases × 30s to 10 phases × 60s. Phase content for M1.4 keeps `basic_walker` only (we'll add variety in M1.5+). Spawn rates can ramp linearly: 0.5/s at phase 0, ~3/s at phase 9. Last phase ends at t=600s.

### Game.run_state.is_over flag (timer freeze on run end)

`Game._process` currently increments `run_state.time_elapsed` every frame unconditionally. After `run_ended` (victory or defeat) the HUD timer should freeze at its final value rather than tick past 10:00 or past the death moment.

Solution: add `var is_over: bool = false` to RunState. `Game.start_run()` resets it to false. `Game._process` early-outs when `is_over == true`. The signal `EventBus.run_ended` fires from somewhere — but to keep things clean, `Game` should subscribe to its own signal and flip the flag, rather than depending on every emit site to also mutate RunState. (Single source of truth: the signal IS the trigger; Game listens.)

```gdscript
# bootstrap/game.gd
func _ready() -> void:
    run_state = RunState.new()
    EventBus.run_ended.connect(_on_run_ended)
    EventBus.run_started.connect(_on_run_started)

func _process(delta: float) -> void:
    if run_state != null and not run_state.is_over:
        run_state.time_elapsed += delta

func _on_run_ended(_victory: bool) -> void:
    run_state.is_over = true

func _on_run_started() -> void:
    run_state.is_over = false
```

### Victory wiring

`SpawnerDirector._on_time_advanced` already detects "schedule complete" and emits `EventBus.wave_completed`. Extend it: when the *final* phase ends (current schedule's `phases.size() - 1`), also emit `EventBus.run_ended(true)`. This signal is currently only used in defeat path; this adds the success path. Arena.gd may want to print a "Victory" log line, but no UI yet (out of scope).

## Edge cases

- **Level-up queue: 3 modals stacked.** If the player picks up a giant XP-gem cluster, the queue could be 3+ deep. Each modal opens immediately after the previous closes — no animation gap, fine for now. UI may feel a bit "popup spam" but matches genre.
- **Modal opens while Player is dead.** `EventBus.level_up` shouldn't fire if Player is dead, but `Game.add_xp` doesn't check — if the player's last gem-pickup-before-death levels them up, the modal opens after death. Add `if Player._is_dead: return` early-out in the picker.
- **Spawner ticks during pause.** `get_tree().paused = true` halts SpawnerDirector's `_process` by default (process_mode INHERIT) — confirm in implementation.
- **Modal during `run_ended(true)`.** If victory fires while a modal is queued, finish the modal queue, THEN process victory. Honor the player's pending picks.
- **Resource-mutation gotcha** (the core architectural guardrail): no upgrade may mutate `WeaponData.tres`. All weapon effects flow through the `effective_*()` read path. Verified by code-review and by the rule that `UpgradeRegistry.apply` only touches Player and HealthComponent for instance-state, and the rest is no-op (consumed at fire time).
- **`run_ended(true)` fires once.** If SpawnerDirector's "last phase complete" check fires across multiple frames, guard with `_victory_emitted: bool = false`.
- **HP bar shows 0 briefly during start_run.** HealthComponent's `_ready` initializes `hp = max_hp`, but the HUD's first `_process` could fire before Player's `_ready`. The HUD's "read from Player" path needs null-safety on the player ref.

### Visual smoke test (windowed, manual)

Per memory `godot_tests_windowed_mode` — Tween / time-driven UI / mouse input need windowed.

1. Run Arena. HUD visible: HP bar full top-left, XP bar empty bottom, level "Lv 1", timer "00:00", weapons list "Blaster Lv.1", FPS bottom-right.
2. Walk around, kill enemies. XP bar fills with each gem. Timer ticks. FPS reads ~60.
3. First level-up: modal appears, game freezes (enemies stop walking, projectiles freeze in mid-air), three cards visible with titles + descriptions.
4. Pick one with mouse click — modal closes, game resumes, the picked stat is visible:
   - +Max HP → HP bar visibly longer
   - +move speed → noticeable speedup
   - +weapon damage → enemies die in fewer hits (eyeball test)
   - +fire rate → projectiles spawn faster (eyeball)
   - Heal-to-full → HP bar refills
5. Level up again — repeat with keyboard 1/2/3. Different key, same outcome.
6. Stack 4-5 level-ups; pick "weapon damage" each time. Enemies should die faster with each pick (compounding evidence).
7. Continue for ~10 minutes (or speed-skip with F-key debug if Debug supports it). At 10:00, console prints `[EventBus] run_ended(victory=true)`, spawner stops emitting, projectiles already in-flight continue, HUD timer **freezes at 10:00** (Game stops incrementing `run_state.time_elapsed` after `run_ended` — see Game-level "is_over" flag below).
8. Die first — make sure death still emits `run_ended(victory=false)` and the modal doesn't open if a posthumous level-up would have queued.

## Risk register

| Risk | Mitigation |
|---|---|
| Pause via `get_tree().paused` breaks SpawnerDirector or Debug logging | Test in headless boot; confirm spawner halts. Modal opts into PROCESS_MODE_WHEN_PAUSED explicitly. |
| `Game.run_state.upgrades_taken` is mutable shared state — any system could append junk | Document that only `UpgradeRegistry.apply` writes. Maybe add an assert checking the upgrade is from the registry's pool. |
| Weapon-effective-value walk perf at scale | M1.4 pool size 5, perf trivial. If pool grows past ~50 OR weapons fire at >100/s, revisit (probably never — VS scales the FUN, not the system count). |
| Modal blocks Player input — what if player tries to pick during pause and a movement key bleeds through? | `get_tree().paused = true` halts Player's `_physics_process`, so movement input does nothing. The modal's input handlers consume number keys via `accept_event()`. |
| Schedule extension to 10 phases hits the M1.2 enemy concurrency cap (30) during late phases | If cap is hit, spawner just pauses — natural difficulty cap. M1.5 will revisit if late-game feels sparse. |
| Weapon damage stacking compounds too hard (multiplicative 5×1.25 = 3.05) | Tunable in `.tres` (drop weight, raise pick count needed). Visual smoke will surface. |
| `UpgradeRegistry` autoload accesses scene tree before Player is ready | Apply path is called from modal interaction, well after `_ready`. Pick_random doesn't touch scene tree. Safe. |
| Modal queue spam (4+ level-ups in a frame from a giant XP burst) | Cap not enforced in M1.4 — if it ever becomes a problem we'll batch. Probably never happens in practice with 1-XP gems. |
| First modal opens before Player.HealthComponent is ready (start_run timing) | First level-up requires XP, which requires gem pickup, which requires player movement — all post-_ready. Safe. |
