# Milestone 1.4 — HUD + Level-up Modal + Upgrade System Foundations

**Date:** 2026-05-17
**Status:** approved (brainstorming complete)
**Predecessor:** [milestone-1.3 player-hp](2026-05-17-milestone-1.3-player-hp.md)
**Parent design:** [vampire-survivors-clone-design](2026-05-16-vampire-survivors-clone-design.md)

## Goal

Make the game playable as a *complete short run*. Surface invisible state via a HUD so the player can see their HP, XP, level, time, weapons. Wire `EventBus.level_up` into a 3-card upgrade-pick modal that pauses the game, lets the player pick 1 of 3 random upgrades, and applies the upgrade by appending to `RunState.upgrades_taken`. Extend the spawn schedule to a 10-minute run with 1-minute phase ramps. Emit `EventBus.run_ended(true)` when the schedule completes.

## Scope

In:
- HUD (5 elements): HP bar, XP bar, level number, run timer (mm:ss), weapons list (text), FPS readout.
- Level-up modal: triggered by `EventBus.level_up`, pauses game, shows 3 random upgrades, click-or-key to pick.
- Upgrade data system: `UpgradeData` Resource, 5 `.tres` files (4 stats + 1 heal), `UpgradeRegistry` autoload, pure-functional application via `RunState.upgrades_taken`.
- `WeaponInstance.effective_damage()` / `effective_fire_rate()` walk `RunState.upgrades_taken`.
- Player `max_hp` / `move_speed` upgrades mutate Player + HealthComponent directly (per-instance state is fine to mutate; data templates are not).
- Spawn schedule extended to 10 phases × 60s = 10-min full run.
- `EventBus.run_ended(true)` emitted by SpawnerDirector at schedule end.

Out (deferred):
- Weapon icons (placeholder text "Blaster Lv.1" only).
- Game Over / Victory UI screens — `run_ended` just fires; debug.gd logs it.
- Restart button.
- Upgrade weight *tuning* — pick honors `UpgradeData.weight` (field shipped in M1.4), but every .tres starts at the default `1.0` so pick is effectively uniform. Designers can rebalance in M1.5+ by editing .tres weights without code changes.
- Mouse-hover tooltips with extended descriptions.
- Reroll / skip / banish upgrade slot mechanics.
- Save / load (Phase 2).
- Audio.
- Theming — Godot defaults for all UI nodes.

## Key decisions (locked in brainstorming)

| # | Decision | Choice |
|---|---|---|
| 1 | Scope | HUD + level-up modal (both, bundled milestone) |
| 2 | HUD elements | All 5: HP / XP / level / timer / weapons-list (text) / FPS |
| 3 | Upgrade pool | 4 stat upgrades (+max HP, +move speed, +weapon damage, +fire rate) + 1 heal-to-full |
| 4 | Pool repeats | Allowed (compounding stacks — standard VS) |
| 5 | Upgrade architecture | Option 3: pure-functional. `RunState.upgrades_taken` is the source of truth; weapons compute effective values from base + upgrade list at fire time. Player stats mutate the Player directly (per-instance), but the upgrade entry still goes in the list for audit. |

| 7 | Multi-level queue | Queued back-to-back. One modal per level crossed. |
| 8 | Run length | 10 min / 10 phases × 60s ramps |
| 9 | Victory emit | `EventBus.run_ended(true)` from SpawnerDirector when schedule's last phase completes |

## Architecture

### Data flow at fire time (read path)

```
WeaponHost._physics_process(delta):
  for weapon in weapons:
    weapon.tick(delta)
    if weapon.can_fire():
      weapon.reset_cooldown()        # uses effective_fire_rate()
      spawn projectile with damage = weapon.effective_damage()

WeaponInstance.effective_damage():
  d := data.base_damage
  for upgrade in Game.run_state.upgrades_taken:
    if upgrade.id == &"weapon_damage_25":
      d *= 1.25
  return d
```

The walk happens once per shot. At realistic pool sizes (5 upgrades in M1.4, ~30 in late-game) this is a few microseconds per call — perf-irrelevant. Always reads current state; no stale multiplier fields to keep in sync.

### Data flow on level-up (write path) — **queue-driven, Side A**

The naive shape would have `Game.add_xp` synchronously increment level N times in one call, emitting `level_up` N times, then leave the picker to drain a queue. That works for M1.4 (no upgrade affects XP math) but bakes in pick-order ambiguity for any future upgrade that does. **Pick-driven instead:** `add_xp` checks the threshold ONCE per call. After each upgrade application, the picker re-asks `Game` whether another level-up is pending. The level only advances *between modal closes*, so the modal always shows accurate state and upgrades that affect XP math apply in pick-order.

```
1. Player picks up XpGem → Game.add_xp(value):
     run_state.xp += value
     EventBus.xp_gained.emit(value)
     Game._maybe_emit_level_up()           # idempotent — emits once if threshold crossed

2. Game._maybe_emit_level_up():
     if run_state.xp >= xp_needed(run_state.level):
       run_state.xp -= xp_needed(run_state.level)
       run_state.level += 1
       EventBus.level_up.emit(run_state.level)
     # else: no-op

3. UpgradePicker._on_level_up(new_level):
     # No queue needed — at most ONE level-up is in flight at a time now.
     if Player._is_dead: return
     _open_modal()

4. UpgradeModal opens:
     get_tree().paused = true               (modal has process_mode = PROCESS_MODE_WHEN_PAUSED)
     options = UpgradeRegistry.pick_random_3()
     render 3 cards
     focus first card                       (keyboard navigation)

5. Player picks card N (mouse click OR 1/2/3 OR Enter on focused):
     upgrade = options[N]
     UpgradeRegistry.apply(upgrade, player) # mutates Player/HealthComponent for stat upgrades; no-op for weapon ones
     run_state.upgrades_taken.append(upgrade)
     EventBus.upgrade_applied.emit(upgrade) # NEW signal; Game listens
     close modal

6. Game._on_upgrade_applied(_upgrade):
     Game._maybe_emit_level_up()            # drains the next pending level if applicable
     if no level_up emitted: get_tree().paused = false
```

Net effect: levels advance one at a time, modals open one at a time, picks apply in order. No queue depth, no race over `run_state.level` value at modal-show time.

`UpgradeRegistry.apply` mutates per-instance state (Player.speed, HealthComponent.max_hp, HealthComponent.hp). Weapon-related upgrades are no-op at apply time — they take effect via the `effective_*()` read path that walks `upgrades_taken`. This means weapon upgrades are "instant" without touching any weapon code at apply.

### Why `UpgradeRegistry` is an autoload (not a Node)

Same reason as `Game` and `EventBus`: it's a global query surface (pick_random anywhere; apply from the modal). No per-scene state. Owns the loaded UpgradeData pool. Autoload count goes from 3 to 4 — within the project's approved list (rubber-duk knowledge base would flag a 5th without justification).

### UpgradeData shape

```gdscript
class_name UpgradeData
extends Resource

@export var id: StringName              # stable id for switch dispatch ("max_hp_20")
@export var display_name: String        # "+20 Max HP (+heal)"
@export var description: String         # one-line flavor / detail shown on card
@export_range(0.0, 10.0) var weight: float = 1.0   # weighted random; default 1.0 across all upgrades → uniform
```

`UpgradeRegistry.apply(upgrade, player)` is a `match upgrade.id:` switch. Adding upgrade #20 in M2.7 = drop a `.tres` + add one `match` branch. Simple linear scaling.

### Pause mechanism + input discipline

`get_tree().paused = true` halts every node by default. The modal opts back in:

```
[node name="UpgradeModal" type="Control" ...]
process_mode = 3   # PROCESS_MODE_WHEN_PAUSED
```

Documented Godot pattern. Projectiles-in-flight, enemies-mid-walk, the spawner, and the Player all freeze cleanly. On modal close, `paused = false` and physics resumes from exactly where it was.

**Input discipline (must spec; otherwise modal silently breaks under windowed play):**

1. Modal `Control` root: `process_mode = WHEN_PAUSED`, `mouse_filter = STOP`.
2. Container ancestors of the card buttons (`CenterContainer`, `PanelContainer`, `HBoxContainer`): `mouse_filter = PASS` so clicks reach the buttons underneath.
3. Card `Button` nodes: `mouse_filter = STOP` (default).
4. After dispatching a pick (mouse click OR keyboard), the modal's input handler calls `get_viewport().set_input_as_handled()` to prevent the same event (especially `ui_accept`/Enter) from bubbling to other listeners — a documented footgun where Enter triggers both "pick" and "unpause" in stacked pause UIs.
5. The picker subscribes to `_unhandled_key_input(event)` for the `1`/`2`/`3` shortcut keys; if `event.keycode in [KEY_1, KEY_2, KEY_3]` and event.pressed, dispatch the pick + `accept_event()`.
6. First card auto-focused on modal open via `card_button.grab_focus()` so Tab/arrow navigation + Enter works without a click.

**Signals during pause (Godot gotcha):** `body_entered`/`area_entered` on paused nodes do NOT fire (Godot issue #47326). This is actually what we want — frozen Player can't pick up XP or take damage mid-modal. But it means `Game.add_xp` cannot be re-entered from a gem pickup during pause (the modal-during-modal scenario is structurally impossible, not just queue-managed).

### HUD subscriptions — **pure signal-driven, no Player reference**

The HUD never reaches into Player's private members. Every visible value is fed by a signal, with two genuinely time-continuous exceptions (run timer, FPS) that read from autoload/engine APIs that don't have signals to subscribe to.

```
HUD._ready():
  EventBus.run_started.connect(_on_run_started)               # reset visible state to defaults
  EventBus.run_ended.connect(_on_run_ended)                   # tint/banner "Run Over"
  EventBus.player_health_changed.connect(_on_player_hp)       # NEW signal — payload (hp, max_hp)
  EventBus.xp_gained.connect(_on_xp_gained)                   # XP bar
  EventBus.level_up.connect(_on_level_up)                     # level label
  EventBus.upgrade_applied.connect(_on_upgrade_applied)       # refresh weapons-list (weapon.level() is derived)

HUD._process(_delta):
  _timer_label.text = format_mm_ss(Game.run_state.time_elapsed)   # Game owns time, no per-tick signal worth emitting
  _fps_label.text = "%d fps" % Engine.get_frames_per_second()     # engine value, no signal exists
```

`EventBus.player_health_changed(hp: float, max_hp: float)` is a NEW signal. It fires from two sources:
- `HealthComponent.damaged` handler in Player (after `_health.take_damage`): re-emit as `(new_hp, max_hp)`
- `HealthComponent.max_hp_changed` handler in Player (after UpgradeRegistry mutates max_hp via set_max_hp): re-emit as `(hp, new_max)`

Player owns this re-emit because it's the canonical "Player's HP changed" event; HealthComponent is a generic component and shouldn't know it lives on Player. (Enemy HealthComponents stay quiet on this bus.)

Net result: HUD has zero `player._health` references, zero `_process` polling of node state, no null-safety edge case on Player ref. Decouples HUD entirely from Player's internal structure.

### Weapons list element — **signal-refreshed, no per-frame iteration**

Subscribes to `EventBus.upgrade_applied` (NEW signal — see Architecture§"New components"). On each pick, the HUD asks for the current weapons list **once** and renders one Label per weapon as `"<weapon_name> Lv.<level>"`. The weapon's `level` comes from `WeaponInstance.level()`, a derived getter that counts matching upgrades in `Game.run_state.upgrades_taken` (also defined under New components above). No per-frame poll of `player._weapon_host`; refresh happens exactly when something changes.

For M1.4 with one weapon this renders:
```
Blaster Lv.1
```

For M1.5+ multi-weapon it scales naturally — one Label per weapon.

The Player ref problem (HUD shouldn't import Player) is solved by reading from `get_tree().get_nodes_in_group("player")` inside the refresh, matching the same group-lookup pattern SpawnerDirector uses. The HUD doesn't keep a stored Player reference between refreshes.

### Schedule extension + Run-end invariant

Extend `world/data/wave_01_schedule.tres` from 2 phases × 30s to 10 phases × 60s. Phase content for M1.4 keeps `basic_walker` only (we'll add variety in M1.5+). Spawn rates can ramp linearly: 0.5/s at phase 0, ~3/s at phase 9. Last phase ends at t=600s.

**Run-end invariant (must enforce centrally):** `EventBus.run_ended` fires **at most once per run**. There are two trigger sources:

- Player death: `Player._on_died` → `Game.end_run(false)`
- Schedule end: `SpawnerDirector` when its last phase completes → `Game.end_run(true)`

Without a central guard, the death-then-schedule-end sequence emits both `run_ended(false)` at t=590s AND `run_ended(true)` at t=600s for the same run — death stops Player but Game._process keeps ticking time and SpawnerDirector keeps processing the schedule. The fix routes both through a single guarded site:

```gdscript
# bootstrap/game.gd
var _run_ended_emitted: bool = false

func end_run(victory: bool) -> void:
    if _run_ended_emitted:
        return
    _run_ended_emitted = true
    run_state.is_over = true   # also stops the time accumulator
    EventBus.run_ended.emit(victory)

func _on_run_started() -> void:
    _run_ended_emitted = false
    run_state.is_over = false
```

`Player._on_died` calls `Game.end_run(false)` instead of emitting `run_ended` directly. `SpawnerDirector` does the same in the wave-completed branch:

```gdscript
# spawner_director.gd — inside the existing _wave_ended_emitted block
if not _wave_ended_emitted:
    _wave_ended_emitted = true
    EventBus.wave_completed.emit()
    Game.end_run(true)       # routes through the guard; safe if Player already died
```

Co-located with the existing `_wave_ended_emitted` guard — no new flag on SpawnerDirector. Subsumes the "what if SpawnerDirector keeps spawning after Player dies" concern: spawning continues (enemies still walk), but only one `run_ended` ever fires.

Test: `test_end_run_emits_once_when_death_followed_by_schedule_end` — call `Game.end_run(false)`, then `Game.end_run(true)`, assert exactly one EventBus.run_ended signal captured with `victory=false`.

### Game timer freeze on run end

`Game._process` increments `run_state.time_elapsed` every frame; after `run_ended` (either outcome) the HUD timer should freeze. Subsumed by the Run-end invariant above: `Game.end_run` flips `run_state.is_over = true`, and `Game._process` early-outs:

```gdscript
func _process(delta: float) -> void:
    if run_state != null and not run_state.is_over:
        run_state.time_elapsed += delta
```

No separate `_on_run_ended` subscription needed — `end_run` is the single mutation site.

### Victory wiring

`SpawnerDirector._on_time_advanced` already detects "schedule complete" and emits `EventBus.wave_completed`. Extend it: when the *final* phase ends (current schedule's `phases.size() - 1`), also emit `EventBus.run_ended(true)`. This signal is currently only used in defeat path; this adds the success path. Arena.gd may want to print a "Victory" log line, but no UI yet (out of scope).

## Edge cases

- **Level-up queue: 3 modals stacked.** If the player picks up a giant XP-gem cluster, the queue could be 3+ deep. Each modal opens immediately after the previous closes — no animation gap, fine for now. UI may feel a bit "popup spam" but matches genre.
- **Modal opens while Player is dead.** `EventBus.level_up` shouldn't fire if Player is dead, but `Game.add_xp` doesn't check — if the player's last gem-pickup-before-death levels them up, the modal opens after death. Add `if Player._is_dead: return` early-out in the picker.
- **Spawner ticks during pause.** `get_tree().paused = true` halts SpawnerDirector's `_process` by default (process_mode INHERIT) — confirm in implementation.
- **Modal during `run_ended` (either outcome).** `run_ended` fires from `Game.end_run` and is a one-shot signal; the modal does NOT consume it in M1.4 (no game-over UI yet). If a level-up modal is open when `run_ended` fires (e.g., schedule end happens to land while the player is mid-pick), the player still picks normally, the upgrade applies, the modal closes, `paused` flips false, and the game resumes in its "over" state (timer frozen, debug log shows `run_ended` exactly once). M2.0's game-over screen will override modal behavior when it lands — out of scope here.
- **Resource-mutation gotcha** (the core architectural guardrail): no upgrade may mutate `WeaponData.tres`. All weapon effects flow through the `effective_*()` read path. Verified by code-review and by the rule that `UpgradeRegistry.apply` only touches Player and HealthComponent for instance-state, and weapon-affecting upgrades are no-op at apply time (consumed at fire time via list-walk).
- **Upgrade list mutation during fire-time walk.** `WeaponInstance.effective_damage()` iterates `Game.run_state.upgrades_taken`. If a pick happened mid-iteration the array would mutate during read — but since modal-open means `paused = true` means weapons aren't firing, no walk is in flight when the append happens. Document this invariant with a one-line comment at the apply site.
- **`run_ended` fires once across both outcome paths.** Death and schedule-end both route through `Game.end_run(victory)`, which guards with `_run_ended_emitted`. SpawnerDirector co-locates the call inside the existing `_wave_ended_emitted` block — no second flag added there.
- **HP bar shows 0 briefly during start_run.** HealthComponent's `_ready` initializes `hp = max_hp`, but the HUD's first `_process` could fire before Player's `_ready`. The HUD's "read from Player" path needs null-safety on the player ref.

## Risk register

| Risk | Mitigation |
|---|---|
| Pause via `get_tree().paused` breaks SpawnerDirector or Debug logging | Test in headless boot; confirm spawner halts. Modal opts into PROCESS_MODE_WHEN_PAUSED explicitly. |
| `Game.run_state.upgrades_taken` is mutable shared state — any system could append junk | Document that only `UpgradeRegistry.apply` writes. Maybe add an assert checking the upgrade is from the registry's pool. |
| Weapon-effective-value walk perf at scale | M1.4 pool size 5, perf trivial. If pool grows past ~50 OR weapons fire at >100/s, revisit (probably never — VS scales the FUN, not the system count). |
| Modal blocks Player input — what if player tries to pick during pause and a movement key bleeds through? | `get_tree().paused = true` halts Player's `_physics_process`, so movement input does nothing. The modal's input handlers consume number keys via `accept_event()`. |
| Schedule extension to 10 phases hits the M1.2 enemy concurrency cap (30) during late phases | If cap is hit, spawner just pauses — natural difficulty cap. M1.5 will revisit if late-game feels sparse. |
| Weapon damage stacking compounds too hard (multiplicative 5×1.25 = 3.05) | Tunable in `.tres` (drop weight, raise pick count needed). Visual smoke will surface. |
| `UpgradeRegistry` autoload accesses scene tree before Player is ready | Apply path is called from modal interaction, well after `_ready`. Pick_random doesn't touch scene tree. Safe. |
| Modal queue spam (4+ level-ups in a frame from a giant XP burst) | Cap not enforced in M1.4 — if it ever becomes a problem we'll batch. Probably never happens in practice with 1-XP gems. |
| First modal opens before Player.HealthComponent is ready (start_run timing) | First level-up requires XP, which requires gem pickup, which requires player movement — all post-_ready. Safe. |
