# Milestone 1.5 — Weapon Variety + Per-Weapon Upgrades

**Date:** 2026-05-17
**Status:** draft (brainstorming)
**Predecessor:** [milestone-1.4 HUD + level-up](2026-05-17-milestone-1.4-hud-and-levelup.md)
**Parent design:** [vampire-survivors-clone-design](2026-05-16-vampire-survivors-clone-design.md)

## Goal

Turn the same-y combat loop into something with build identity. Add **3 new weapons** with distinct mechanics (Spread, Aura, Orbital) acquired via the M1.4 level-up modal. Each weapon ships with **2 per-weapon upgrades** in the pool. Existing `weapon_damage_25` / `fire_rate_30` upgrades are split per-weapon so picks always tie to something the player owns. By end of run the player can plausibly own 1–4 of the 4 total weapons and have stacked multipliers on the ones they kept.

## Scope

In:
- **3 new weapons:** Spread Shot, Aura, Orbital Blade. Plus Blaster (existing) = 4 weapons total.
- **Weapon-scene-per-kind architecture:** `WeaponData.weapon_scene: PackedScene` field. WeaponHost instantiates the scene as a Player child on acquisition. Per-weapon behavior lives in that scene's script.
- **Per-weapon upgrade split:** `weapon_damage_25` / `fire_rate_30` are removed and replaced with per-weapon equivalents (`blaster_damage_25`, `aura_damage_25`, etc.). WeaponInstance computes effective values via `data.id`-keyed lookup.
- **Weapon-specific mechanical upgrades:** `aura_radius_25`, `orbital_count_plus_1`, `spread_pellets_plus_1`. Each weapon gets exactly 2 upgrades (1 damage-style + 1 mechanical-style).
- **Acquisition cards:** `acquire_aura`, `acquire_orbital`, `acquire_spread`. Picked from the modal pool when the player doesn't own that weapon.
- **Tiered pool gating:** `UpgradeRegistry.pick_random_3()` filters the pool by ownership state at pick time. Un-owned weapon-upgrade cards are hidden until acquisition. Already-acquired weapon-acquisition cards are hidden after acquisition.
- **HUD WeaponList scales** to N weapons automatically (already signal-driven from M1.4; no changes needed beyond verifying it scales).

Out (deferred):
- Weapon icons (text-only `"<name> Lv.N"` continues).
- Per-weapon balance tuning beyond shipping defaults — designers can rebalance via `.tres` weight edits in M1.6+.
- Reroll / skip / banish slot mechanics (still M2.x).
- Player-loadout-select pre-run (no menu surface in M1.5; first run still starts with Blaster only).
- More than 2 upgrades per weapon (per-weapon "tier-up" / evolution is a future milestone topic).
- Audio.

## Architecture

### Weapon-scene-per-kind: the central refactor

Today's WeaponHost assumes every weapon is "fire a single projectile on cooldown" — works for Blaster, but Aura (persistent damage zone) and Orbital (orbiting blade) have entirely different lifecycles. Three options were considered (see `docs/plans` history); the cleanest is to push the per-kind logic into per-kind scenes:

```
WeaponData (Resource):
  id: StringName             — e.g. &"aura"; used for upgrade-lookup keys
  display_name: String       — "Aura"
  base_damage: float
  fire_rate: float           — for cooldown-firing weapons; ignored for persistent ones
  range: float               — for projectile-firing weapons; ignored for persistent ones
  projectile_scene: PackedScene  — kept for Blaster/Spread compatibility
  weapon_scene: PackedScene  — NEW: the scene spawned under Player; nullable
```

```
WeaponInstance (RefCounted, thin):
  data: WeaponData
  node: Node                 — NEW: the spawned weapon_scene instance (nullable)
  cooldown_remaining: float  — kept for cooldown-firing weapons
  + effective_damage()       — walks upgrades_taken for "<id>_damage_25"
  + effective_fire_rate()    — walks upgrades_taken for "<id>_fire_rate_30"
  + level()                  — derived: 1 + count of weapon-affecting upgrades matching this weapon
  + count_upgrade(id)        — generic helper for mechanical upgrades; counts matches in upgrades_taken
```

```
WeaponHost (Node, child of Player):
  add_weapon(data):
    instance := WeaponInstance.new(data)
    if data.weapon_scene != null:
      instance.node = data.weapon_scene.instantiate()
      get_parent().add_child(instance.node)  # Player owns it
      if instance.node has 'configure': instance.node.configure(instance)
    weapons.append(instance)
    refresh hud weapon list (via existing EventBus.upgrade_applied path)

  _physics_process(delta):
    for w in weapons:
      if w.node != null and w.node has '_owned_tick': w.node._owned_tick(delta)
      else: existing cooldown-fire path for Blaster
```

The split: **scene-owned weapons** (Aura, Orbital) handle their own behavior inside the spawned node; **cooldown-fire weapons** (Blaster, Spread) keep the existing WeaponHost-driven fire-on-cooldown path. WeaponHost branches on whether the weapon scene exposes a tick method.

(Blaster could optionally migrate to a weapon-scene later for consistency; for M1.5 we keep its existing path to minimize risk to the just-shipped M1.4 visual smoke.)

### The 3 new weapons

#### Spread Shot — cone of projectiles

- Cooldown-fire (same model as Blaster). On fire, instantiates N `BasicProjectile`s in a cone toward nearest enemy.
- Default: 3 pellets, ±15° cone, fire_rate 0.8/s.
- Reuses `combat/weapons/projectiles/basic_projectile.tscn`. No new projectile asset needed.
- Per-weapon upgrades: `spread_damage_25` (+25% pellet damage), `spread_pellets_plus_1` (+1 pellet per shot).
- No weapon_scene needed; WeaponHost handles it via the existing cooldown-fire path with a `targeting: Targeting.SPREAD` field added to WeaponData enum (or a new field `pellet_count: int = 1`; cleaner). Pellet count = `1 + count_upgrade(&"spread_pellets_plus_1")`.

#### Aura — persistent damage zone around Player

- Weapon scene: `combat/weapons/aura/aura_weapon.tscn`. Area2D child of Player, CircleShape2D collision, ticks damage every N seconds to overlapping enemies (filtered by `enemies` group).
- Default: 80px radius, 1.0 damage every 0.5s.
- Per-weapon upgrades: `aura_damage_25` (+25% per-tick damage), `aura_radius_25` (+25% radius).
- AuraWeapon script queries `Game.run_state.upgrades_taken` directly each tick for effective radius + damage (consistent with WeaponInstance's pure-functional read path).
- Re-uses M1.3 collision layer scheme: aura's Area2D `collision_mask = ENEMIES (4)`.

#### Orbital — N blades orbiting Player

- Weapon scene: `combat/weapons/orbital/orbital_weapon.tscn`. Node2D child of Player owning N `OrbitalBlade` children (Area2D + Polygon2D triangle). Blades positioned by rotation around Player at fixed radius.
- Default: 1 blade, 60px orbit radius, full rotation in 1.5s, 5 damage per touch with re-hit cooldown (per-enemy) of 0.5s.
- Per-weapon upgrades: `orbital_damage_25` (+25% per-touch damage), `orbital_count_plus_1` (+1 blade).
- OrbitalWeapon._process(delta) updates rotation; on `orbital_count_plus_1` count change, spawns/frees blade children to match.
- Re-hit cooldown is per-enemy (Dictionary keyed by enemy instance_id, decremented per tick) — avoids one-frame multi-hit while keeping orbit gameplay readable.

### Per-weapon upgrade key scheme

Upgrades use `<weapon_id>_<kind>_<value>` ids:

| Upgrade | Id |
|---|---|
| Acquire Aura | `acquire_aura` |
| Acquire Orbital | `acquire_orbital` |
| Acquire Spread | `acquire_spread` |
| Blaster +damage | `blaster_damage_25` |
| Blaster +fire rate | `blaster_fire_rate_30` |
| Spread +damage | `spread_damage_25` |
| Spread +pellets | `spread_pellets_plus_1` |
| Aura +damage | `aura_damage_25` |
| Aura +radius | `aura_radius_25` |
| Orbital +damage | `orbital_damage_25` |
| Orbital +blade count | `orbital_count_plus_1` |

UpgradeRegistry.apply dispatch grows from 5 arms to 14 (8 weapon + 3 acquire + 3 stat). Acquire arms call `WeaponHost.add_weapon(load(data_path))`. Damage/fire-rate arms are no-op at apply time (consumed at fire/tick time via WeaponInstance methods). Mechanical arms (radius_25, pellets_plus_1, count_plus_1) are also no-op at apply time and consumed by the weapon-scene scripts themselves at runtime via `count_upgrade()`.

The `_:` push_warning fallthrough from M1.4 catches any forgotten match arm.

### Pool gating

`UpgradeRegistry.pick_random_3()` filters the master pool to "available" before sampling:

```gdscript
func pick_random_3() -> Array[UpgradeData]:
    var available := _available_for_picker()
    # existing weighted-without-replacement sampling on `available`
    ...

func _available_for_picker() -> Array[UpgradeData]:
    var owned_weapon_ids := _owned_weapon_ids()
    var taken_ids := _taken_upgrade_ids()
    var result: Array[UpgradeData] = []
    for u in pool:
        if _is_weapon_acquisition(u) and _weapon_id_for_acquisition(u) in owned_weapon_ids:
            continue  # already own it
        if _is_weapon_specific(u) and not (_weapon_id_for_upgrade(u) in owned_weapon_ids):
            continue  # gated until acquired
        if _is_one_shot(u) and u.id in taken_ids:
            continue  # M1.5 doesn't have any (heal_to_full is repeatable); reserve hook for future
        result.append(u)
    return result
```

`_owned_weapon_ids()` reads `WeaponHost.weapons` via the "player" group; `_taken_upgrade_ids()` walks `Game.run_state.upgrades_taken`. Both are O(N) but N is small (max 17). Run-time cost negligible.

Classification helpers (`_is_weapon_acquisition`, `_is_weapon_specific`, `_weapon_id_for_*`) read from the UpgradeData's `id` StringName by prefix-stripping:

```gdscript
func _is_weapon_acquisition(u: UpgradeData) -> bool:
    return String(u.id).begins_with("acquire_")

func _weapon_id_for_acquisition(u: UpgradeData) -> StringName:
    return StringName(String(u.id).trim_prefix("acquire_"))   # "aura"

func _is_weapon_specific(u: UpgradeData) -> bool:
    var s := String(u.id)
    for w in [&"blaster", &"aura", &"orbital", &"spread"]:
        if s.begins_with(String(w) + "_"):
            return true
    return false

func _weapon_id_for_upgrade(u: UpgradeData) -> StringName:
    var s := String(u.id)
    return StringName(s.split("_")[0])
```

Yes, this is string-prefix sniffing. Trade-off: it keeps the UpgradeData shape unchanged (single `id` field, no taxonomy enums to drift), at the cost of a slim coupling between id-naming conventions and registry logic. Documented invariant: **upgrade ids MUST use the `<weapon_id>_<kind>_<value>` or `acquire_<weapon_id>` pattern**. Anything else is treated as a stat upgrade (i.e., always available, no gating).

### WeaponInstance effective-value lookups

`WeaponInstance.effective_damage()` was hardcoded to `&"weapon_damage_25"` in M1.4. Now it builds the lookup key from `data.id`:

```gdscript
func effective_damage() -> float:
    if data == null: return 0.0
    var d := data.base_damage
    var damage_key := StringName("%s_damage_25" % String(data.id))
    for upgrade in Game.run_state.upgrades_taken:
        if upgrade.id == damage_key:
            d *= WEAPON_DAMAGE_MULTIPLIER
    return d

func effective_fire_rate() -> float:
    if data == null: return 0.0
    var r := data.fire_rate
    var rate_key := StringName("%s_fire_rate_30" % String(data.id))
    for upgrade in Game.run_state.upgrades_taken:
        if upgrade.id == rate_key:
            r *= FIRE_RATE_MULTIPLIER
    return r

func level() -> int:
    var n := 1
    var prefix := String(data.id) + "_"
    for upgrade in Game.run_state.upgrades_taken:
        if String(upgrade.id).begins_with(prefix):
            n += 1
    return n

func count_upgrade(id: StringName) -> int:
    var n := 0
    for upgrade in Game.run_state.upgrades_taken:
        if upgrade.id == id:
            n += 1
    return n
```

The existing 2 `.tres` (`weapon_damage_25.tres`, `fire_rate_30.tres`) are deleted; 8 new per-weapon equivalents replace them. Existing M1.4 weapon-instance tests update to use `blaster_damage_25` keys.

## Edge cases

- **Player owns all 4 weapons.** Pool stops surfacing acquire cards; all 8 weapon-specific upgrades are eligible. Player still has 3 stat cards in mix. Pool size 11; no problem.
- **Player has 0 acquire cards but pool is small.** If only 2 stat + 1 weapon-specific cards are eligible (rare; requires owning Blaster only and having taken all 3 stat picks), `pick_random_3()` returns ≤3. Existing degrade path handles it.
- **Multi-weapon `_refresh_weapon_list` during pause.** HUD's `_refresh_weapon_list` runs on `EventBus.upgrade_applied`, which fires synchronously inside the picker's `_apply_pick` (still inside the modal's input chain). Signal handlers fire even when the tree is paused; `_process` does not. The refresh runs cleanly. Verified safe.
- **Aura on dead player.** Aura's `_owned_tick` should early-out if Player `_is_dead`. Aura is a Player child; when Player dies, `set_physics_process(false)` halts the WeaponHost's tick loop, which in turn means Aura's `_owned_tick` isn't called. Confirm in implementation. (For consistency with Blaster's death behavior, which stops firing.)
- **Orbital on dead player.** Same — WeaponHost halts. Orbital blades freeze in mid-rotation. Visually fine; matches the "everything stops" death feel.
- **Acquiring a weapon mid-modal during multi-level chain.** The acquisition picks `acquire_aura`, which calls `WeaponHost.add_weapon(aura_data)`. That instantiates AuraWeapon as a Player child. Player is still paused (modal A's pick → emit chain → modal B opens before unpause). Aura's `_owned_tick` will fire on next physics frame after unpause. Safe.
- **`count_upgrade` re-walk perf.** Each Aura tick (every 0.5s) walks `upgrades_taken` once. Orbital walks every frame for rotation but only walks `upgrades_taken` on count change (or every frame if we want it live — tunable). Spread walks once per shot. Same negligible perf as M1.4.

Automated:
- All M1.4-era tests pass.
- New per-weapon test suites pass.
- `godot --headless --quit --path .` exits 0.

- All 4 weapons functional and distinct.
- Pool gating works (no `Acquire: X` after acquiring X; no `X +damage` before acquiring X).

## Risk register

| Risk | Mitigation |
|---|---|
| Late-game throughput throttled at 30 concurrent enemies (inherited from M1.2) | M1.4 phases F-J spec 2.0-3.0 spawns/sec, but the cap saturates around phase F. M1.5 includes a `max_concurrent_enemies` revisit: bump to 60 + add a quick perf sanity-check in the visual smoke. If FPS holds at 60, ship; if not, flatten the late-phase schedule to ~2.0/s and document the cap as the intended difficulty wall. Rubber-duk caught the spec-vs-runtime mismatch in the M1.4 review. |
| Weapon-scene-per-kind refactor breaks Blaster's M1.4 visual smoke | Blaster keeps existing cooldown-fire path; only Aura/Orbital go through weapon_scene. Tests + smoke verify Blaster behavior unchanged. |
| Aura ticks too often / kills FPS at high spawn rates | Default tick rate 0.5s with `Engine.get_frames_per_second()` HUD readout already in place. Visual smoke catches regressions. |
| Orbital re-hit cooldown leaks Dictionary entries on dead enemies | On enemy_killed signal, the orbital's cooldown Dict gets cleaned. Or: TTL-based eviction (entries older than 2× cooldown are dropped). |
| Pool gating logic mis-classifies a future custom upgrade id | Documented invariant: ids use `<weapon_id>_<kind>_<value>` or `acquire_<weapon_id>` or are stat upgrades. Anything else slots into stat-upgrade tier (always available). Documented in `upgrade_registry.gd` header. |
| Splitting `weapon_damage_25` → 4 per-weapon variants makes early-game weak | At level 2-3 (when pool is mostly Blaster-only), all 3 acquires + 2 blaster upgrades + 3 stat = 8 cards available. Reasonable variety. |
| WeaponInstance.level() prefix-match collides with stat upgrade ids | Stat ids (`max_hp_20`, etc.) don't start with weapon ids. The "_" terminator in the prefix-check rules out partial matches (e.g., `blaster_` won't match `blastersomething`). Verified by test. |
| `count_upgrade` over `upgrades_taken` walk perf with 50+ upgrades | M1.5 pool capped at 17. M2.x can revisit if pool grows past 50 by caching counts in RunState (not needed yet). |
| Acquire card surfaces when modal is open and Player can't pause-instantiate the weapon scene | Modal-pick → apply runs synchronously while paused. `WeaponHost.add_weapon` does `get_parent().add_child(instance.node)`, which is safe under pause. Aura/Orbital `_ready` does NOT depend on tree state. Verified by M1.4 pause infra. |
