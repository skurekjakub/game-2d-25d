# Game Programming Patterns — local mirror

A local markdown mirror of [Game Programming Patterns](https://gameprogrammingpatterns.com/)
by Robert Nystrom. Mirrored for personal study and as a grounding reference for
agent-assisted work on this codebase. Not redistributed.

If you find it useful, **buy the book** — Robert maintains it free online but
the print/ebook versions support him directly:
https://gameprogrammingpatterns.com/

Generated by `/tmp/mine_gpp.sh` from the live site. Each file has a header
linking back to the original chapter URL. Images aren't mirrored — references
in the text point at relative `images/*.png` paths that won't resolve locally;
view the source URL if you need a diagram.

## Design Patterns Revisited

The Gang of Four classics, re-evaluated through a game-development lens. Read
these first if you're already familiar with the original GoF patterns.

- [Command](command.md) — encapsulate a request as an object; undo/redo, input remapping, AI orders.
- [Flyweight](flyweight.md) — share fine-grained state across many objects (forests, particle systems).
- [Observer](observer.md) — broadcast events without coupling source to receivers (achievements, UI updates).
- [Prototype](prototype.md) — clone objects as templates; data-driven entity definitions.
- [Singleton](singleton.md) — Nystrom's anti-pattern chapter: why most singletons in games are wrong.
- [State](state.md) — finite state machines; hierarchical, pushdown, and concurrent variants.

## Sequencing Patterns

How time and order get expressed in code.

- [Double Buffer](double-buffer.md) — present a coherent frame while writing the next one (render, physics).
- [Game Loop](game-loop.md) — the master clock; fixed vs variable timestep, decoupled physics.
- [Update Method](update-method.md) — simulate each entity one frame at a time.

## Behavioral Patterns

How game behavior gets defined and composed, especially when designers (not
programmers) need to author it.

- [Bytecode](bytecode.md) — design a tiny scripting language for designers; data-driven AI.
- [Subclass Sandbox](subclass-sandbox.md) — base class provides primitives, subclasses combine them.
- [Type Object](type-object.md) — define types as data (designer-authored) instead of as classes.

## Decoupling Patterns

Reduce coupling so subsystems can evolve independently.

- [Component](component.md) — split one monolithic class into a bag of single-purpose components (ECS lite).
- [Event Queue](event-queue.md) — decouple sender and receiver in time as well as code.
- [Service Locator](service-locator.md) — global access without the singleton trap.

## Optimization Patterns

CPU and cache-friendliness — when "make it work" needs to become "make it fast".

- [Data Locality](data-locality.md) — pack hot data contiguously so the CPU cache stays warm.
- [Dirty Flag](dirty-flag.md) — defer work until the result is actually needed.
- [Object Pool](object-pool.md) — reuse instances instead of allocating; bullets, particles, audio.
- [Spatial Partition](spatial-partition.md) — grid / quadtree / BSP for O(1) neighbour queries.

## How to use this with the project

These chapters are most useful as **pre-design grounding** before writing a new
system. Two patterns from this book are already load-bearing in the codebase:

| Repo pattern | Book chapter |
|---|---|
| `UpgradeEffect` Strategy (Bundle 2 of pre-M1.6 refactor) | [State](state.md) is closest cousin (typed effect class per variant); also relates to GoF Strategy not covered here |
| `EventBus` signal hub | [Observer](observer.md) directly |
| `WeaponInstance.tick(delta)` + `_owned_tick(delta)` | [Update Method](update-method.md) directly |
| `UpgradeData` shared resource + `WeaponInstance.effective_*` walk | [Type Object](type-object.md) directly |
| `Game.run_state` autoload, `Player` group lookup | [Service Locator](service-locator.md) and [Singleton](singleton.md) (anti-pattern) |
| `Damageable.try_damage` static helper | Not a book pattern — DRY refactor only |
| Future: M1.6 `DamageAggregator` autoload, `RunStats` Resource | [Observer](observer.md) + [Dirty Flag](dirty-flag.md) if we cache aggregates |
| Future: `BasicProjectile` lifecycle reuse | [Object Pool](object-pool.md) — relevant when projectile churn becomes a profile hit |
| Future: spatial neighbour queries for swarm AI | [Spatial Partition](spatial-partition.md) |

If you're about to design something new, skim the relevant chapter first. The
worst design discussions in this repo's history were the ones where neither
party had read Nystrom's take on the same problem.
